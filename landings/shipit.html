Three steps to blast-off

From a file to a live link.

Your agent writes a file, you run one command, and getonup ships it into orbit. No dashboard wrangling, no viewer login.

1

Your agent builds it

The coding agent writes a .html, .tsx, or .vue. A single bare component is fine, with no boilerplate and no bundler config.

2

Run getonup deploy

One command wraps the component into a self-contained page and boosts it straight to your bucket. That's the blast-off.

3

It lands in orbit

You get a clean URL on your own domain that scales to zero. Share it, screenshot it, or embed it, and it stays live without a server running.

What you get

A tiny stack with serious thrust.

Free software, MIT licensed, with every piece running inside your Cloudflare account.

~$0 idle

One Worker plus R2, scale-to-zero

A single Cloudflare Worker serves your artifacts and one R2 bucket stores them. There's no database and no always-on box to keep warm, so it idles all the way down to roughly nothing and only spins up when someone opens a link.

cloudflare worker + r2

Agent-native

Drop a snippet in AGENTS.md and any coding agent can publish on its own. It's one CLI call any model already knows how to make, with no SDK and no auth dance.

drop-in for agents

Auto-wrap, no build step

Hand it one bare .tsx, .vue, or .html and it becomes a runnable page. React 18 with Babel and Tailwind, or Vue 3, gets inlined into one self-contained file.

react 18 · vue 3

Yours to self-host, MIT licensed

Your Cloudflare account, your token, your domain, your data. The whole thing is MIT licensed and self-hostable end to end, so you can fork it, audit it, and run it for as long as you like without asking anyone.
